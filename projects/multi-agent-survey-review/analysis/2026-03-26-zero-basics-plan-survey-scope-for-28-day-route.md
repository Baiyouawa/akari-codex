## 1. 本次综述调研的直接目标

本轮不是做泛化的“零基础学习研究”，而是为 `zero-basics-plan` 的 **28 天课程路线增强** 建立一套可立即执行的综述调研目标定义。依据 `projects/zero-basics-plan/analysis/2026-03-26-28-day-course-outline.md`，当前课程主线已经确定为四段：

1. 远程开发环境与 Linux 入门
2. Linux 实战深化与 Git / GitHub 基础
3. GitHub 协作与 AI 编程工作流
4. 深度学习入门与综合结课

因此，本轮综述调研的目标不是再讨论“零基础应该学什么”的大而全答案，而是回答下面三个直接服务课程设计的问题：

- **课程是否缺少关键主题证据**：哪些模块已经有足够资料，哪些模块仍需补强。
- **课程如何更稳地支撑零基础学习者**：哪些地方需要增加前置解释、练习、复盘、风险提示。
- **哪些外部综述或高质量资料值得继续检索**：后续资料收集任务应把资源投到哪里，才能直接改善 28 天路线。

换言之，本任务的产出应直接服务三个后续环节：

1. `projects/multi-agent-survey-review` 后续“检索并收集可支撑该课程路线的高质量资料与综述来源”任务；
2. `projects/multi-agent-survey-review` 后续“汇总已有最终结果与新检索证据，生成可并入 zero-basics-plan 的综述摘要”任务；
3. `projects/zero-basics-plan` 后续对 28 天教程的扩写、补练习、补复盘、补来源与补系统课步骤工作。

## 2. 28 天课程路线带来的硬约束

### 2.1 时间与节奏约束

依据 `projects/zero-basics-plan/analysis/2026-03-26-28-day-course-outline.md`：

- 总时长固定为 28 天。
- 工作日建议每天 60–120 分钟。
- 周末系统课建议 120–180 分钟。
- Day 7 / 14 / 21 / 28 负责整周串联，而不是引入完全新主线。

这意味着后续综述调研必须优先关注：

- **可在 1–2 小时内完成的学习单元**；
- **能直接对应到某一天或某一周系统课的资料**；
- **能够降低掉队风险的解释、练习与复盘模板**；
- **能支撑“做完即有产出”的任务设计**。

不满足这一时间颗粒度的内容，即便有价值，也不应作为当前主范围优先项。

### 2.2 课程目标约束

依据同一课程大纲，28 天后的最小闭环是：

1. 能使用 VS Code Remote SSH 连接 Linux 主机；
2. 能完成 Linux 常见文件、权限、进程、网络与 SSH 操作；
3. 能使用 Git 与 GitHub 完成本地版本管理、远程同步、分支开发与 Pull Request；
4. 能理解并实践 AI 辅助编程的最小工作流；
5. 能理解深度学习基础概念并跑通一个最小 PyTorch 实验；
6. 能完成一个带仓库记录的结课小项目。

因此，调研目标应围绕“如何更稳地到达上述六项结果”展开，而不是扩展到更远的职业训练或研究训练目标。

### 2.3 当前缺口约束

依据 `projects/zero-basics-plan/analysis/2026-03-26-28-day-gap-scan.md`，当前最主要缺口不是“主线缺失”，而是：

- 练习任务普遍没有独立成段；
- 普通日复盘作业普遍缺失；
- Day 14 缺少独立操作步骤；
- 系统课之间的结构风格尚未完全统一。

这决定了后续综述调研应优先收集：

- 新手友好的实操练习模板；
- 日更复盘问题模板；
- 报错排查流程与验证清单；
- 系统课串联框架，而不是继续大规模发散新主题。

## 3. “零基础”在本项目中的工作性定义

### 3.1 应默认学习者不会什么

依据 `projects/zero-basics-plan/README.md` 与 28 天路线正文，本项目中的“零基础”应解释为：

- 对 Linux 目录、命令、权限、进程、SSH 没有稳定操作经验；
- 对 Git / GitHub 的工作区、提交、分支、PR 没有实践经验；
- 对 Cursor / AI 编程工作流缺乏结构化使用经验；
- 对深度学习中的张量、模型、训练、推理没有成体系概念；
- 对从“本地电脑 → 远程主机 → 仓库协作 → AI 辅助开发 → 最小模型实验”的完整链路没有经历过。

### 3.2 可默认学习者已经具备什么

当前课程大纲已给出“适用对象：能正常使用电脑”的限制，因此可默认学习者具备：

- 基础数字素养：会安装软件、解压文件、复制路径、打开终端或命令提示符；
- 基础英文界面容忍度：能在教程指导下阅读按钮、菜单与常见错误提示；
- 基础文件管理能力：知道文件夹、文件、浏览器、文本编辑器的基本使用；
- 基础网络使用能力：能登录网站、注册 GitHub、复制链接。

### 3.3 明确不纳入当前“零基础”定义的情况

以下情况不应默认纳入本轮课程或调研范围：

- 完全不会使用电脑、不会安装软件、不会键盘输入的绝对数字零基础；
- 需要同时补中学数学、英语基础、通识计算机教育的大范围基础教育问题；
- 需要在 28 天内完成职业化工程训练、团队级开发规范、生产环境部署或复杂模型训练的人群。

### 3.4 由此带来的调研口径

后续调研应优先寻找：

- 对“有通用电脑操作能力但无本领域经验”的学习者最友好的材料；
- 可直接支撑“上手、练习、复盘、排错”的资料；
- 能降低术语理解门槛而不牺牲操作闭环的内容。

而不应把调研范围扩大到整个基础教育、职业培训或专家级路线设计。

### 这些核心主题应如何转化为调研包

后续应优先围绕以下五包资料检索：

1. **学习者与课程结构包**：零基础假设、每天负担、4 周能力跃迁。
2. **练习与复盘包**：练习模板、复盘问题、打卡标准、当日产出规范。
3. **实操与排错包**：SSH、权限、Git、环境配置、PyTorch 安装等报错处理。
4. **AI 工作流包**：Cursor / AI 编程的安全边界、diff 验证、规则约束、最小协作闭环。
5. **结课与迁移包**：结课项目、最终能力证明、后续 30 天进阶模板。

## 5. 结构化问题清单

下列问题清单用于直接指导后续并行调研。每个问题都应能被检索、归纳并映射回课程设计。

## 5.1 学习者与前置假设

1. 当前课程默认的数字素养下界是什么？
2. 哪些能力可作为默认已具备，哪些必须在课程内显式教授？
3. 哪些常见前置坑会让学习者在 Day 1–3 就掉队？
4. 学习者最容易误解的术语有哪些，哪些必须给生活化解释？
5. “能正常使用电脑”这一假设对应的最低操作清单应如何写清楚？

## 5.2 课程目标与最终产出

1. 28 天后最小闭环到底要以“会操作”为主，还是“会解释 + 会操作”为主？
2. Day 27/28 的结课产出最小验收标准是什么？
3. 应该如何定义“完成课程”，才能既不过高也不过低？
4. 哪些产出最能证明学习者已经跨过零基础门槛：仓库、实验记录、复盘、PR、脚本，还是它们的组合？

## 5.3 知识结构与依赖关系

1. 哪些主题必须在前面先学，否则后面无法推进？
2. VS Code Remote SSH、Linux 命令、Git、GitHub、Cursor、PyTorch 之间的依赖链如何最清晰地向新手解释？
3. 哪些天本质上是在学“工具概念”，哪些天是在学“操作技能”，哪些天是在学“工作流”？
4. 当前四周划分是否已经是最小认知负担方案，还是还需要重新排序某些日程？

## 5.4 时间分配与节奏控制

1. 每天 60–120 分钟足够覆盖哪些内容，不足以覆盖哪些内容？
2. 哪些天应明确标为“轻量理解日”，哪些天应标为“重实操日”？
3. 系统课应该承担多少新内容，多少复盘内容？
4. 零基础学习者连续 7 天里合理的失败容忍度和补课机制应如何体现？

## 5.5 实践任务与练习设计

1. 每一天的练习是否都能写成“输入 / 执行动作 / 完成标准 / 提交产物”四段式？
2. 哪些主题最需要截图、命令输出、仓库链接等可验证产物？
3. 哪些练习适合纯本地完成，哪些必须依赖远程 Linux 或 GitHub？
4. Day 14 这类系统课的综合练习应如何拆成可执行实验单？
5. 第 4 周的 PyTorch 实验最小到什么程度才不会压垮零基础学习者？

## 5.6 评估与复盘机制

1. 普通日复盘最少应该问几个问题才足够形成闭环？
2. 系统课复盘应和普通日复盘有哪些差异？
3. 哪些问题能帮助判断“今天只是做过了”还是“真的理解了”？
4. 是否需要统一的每日打卡模板、周复盘模板、结课复盘模板？

## 5.7 风险点与掉队点

1. Day 1–3 最大风险是环境失败还是概念过载？
2. 第 2 周最大风险是权限 / SSH / Git 报错，还是任务密度过高？
3. 第 3 周最大风险是 GitHub 协作概念抽象，还是 AI 工具使用失控？
4. 第 4 周最大风险是环境配置失败，还是深度学习术语劝退？
5. 哪些位置应显式加入“如果失败，先检查这三项”的提示？

## 5.8 资料检索与来源策略

1. 每个章节最需要哪类资料：官方文档、长期维护教程、中文补充、案例、FAQ、报错贴？
2. 哪些主题必须优先官方文档，哪些主题更需要中文生活化材料降低门槛？
3. 哪些链接应作为正文主资料，哪些只适合做附录扩展阅读？
4. 后续资料库是否需要标注“适合入门 / 适合进阶 / 适合排错 / 适合系统课串联”？

## 6. 问题优先级

## 6.1 第一优先级：立即决定课程成败的问题

这些问题必须在首轮并行调研中优先回答。

1. 零基础学习者的前置假设边界是什么？
2. 每一天的完成标准与提交产物应如何统一？
3. 普通日练习与复盘模板如何补齐？
4. Day 14 等系统课的操作步骤如何标准化？
5. SSH / 权限 / Git / PyTorch 安装等高频报错应如何形成统一排查清单？
6. 第 3 周 AI 编程工具使用应如何加上验证与边界提示？

## 6.2 第二优先级：首轮综述后补强的问题

1. 中文社区案例的生活化补充应如何选材？
2. 系统课是否需要更强的图示、概念图、截图材料？
3. 第 4 周深度学习部分是否需要更多“概念→实验→解释”桥接材料？
4. 结课项目是否应区分“开发流程型”和“深度学习体验型”的不同完成标准？

## 6.3 暂缓处理的问题

1. 竞品课程对标与商业化定位；
2. 正式录播、题库、平台化交互设计；
3. 进阶课程、岗位能力图谱、长期社群运营；
4. multi-agent 研究专题如何作为下一阶段课程单独开线。

## 7. 并行综述调研的拆分方式

## 7.1 推荐拆分维度

建议按“课程增强所需的问题包”而不是按周简单切分。这样更利于汇总。

### 调研包 A：学习者画像与前置假设
- 目标：收敛“零基础”的工作性定义。
- 重点问题：前置能力、术语理解门槛、早期掉队点。
- 直接输出：适用于课程首页或导学部分的“适用对象 / 不适用对象 / 先修说明”。

### 调研包 B：日更练习与复盘模板
- 目标：补齐 `28-day-gap-scan.md` 已指出的最大共性缺口。
- 重点问题：练习模板、复盘模板、完成标准、提交产物。
- 直接输出：可批量贴到 Day 1–28 的结构模板。

### 调研包 C：实操报错与验证清单
- 目标：解决零基础最容易卡死的技术节点。
- 重点问题：SSH、权限、Git、解释器、PyTorch 环境。
- 直接输出：排查树、常见错误表、验证命令清单。

### 调研包 D：AI 编程工作流边界
- 目标：明确 Cursor / Agent 如何安全、低风险地引入课程。
- 重点问题：Ask/Edit/Agent 区别、规则、diff 检查、最小验证。
- 直接输出：AI 工具使用守则、练习模式、课堂提示语。

### 调研包 E：结课项目与迁移路线
- 目标：反向定义 Day 27/28 的完成标准与后续 30 天计划结构。
- 重点问题：结课项目类型、验收、复盘、后续进阶。
- 直接输出：结课 rubric、能力清单、后续进阶模板。

为保证并行结果可合并，每个调研包都应统一使用以下字段：

| 字段 | 说明 |
|---|---|
| 研究问题 | 本条在回答什么课程设计问题 |
| 范围说明 | 覆盖哪些天、哪些章节、哪些学习者情境 |
| 关键结论 | 可以直接拿来指导课程的结论 |
| 证据来源 | URL、官方文档页、仓库文件路径 |
| 对 28 天路线的影响 | 影响哪一周、哪一天、哪个系统课 |
| 建议动作 | 直接并入 / 改写后并入 / 作为补充资料 / 暂缓 |
| 待确认项 | 证据不足、需要二轮检索或人工判断的点 |

## 7.3 汇总与冲突消解规则

- 若多个调研包给出相同结论，优先保留可直接落到课程结构的表述。
- 若多个调研包结论冲突，优先采用更贴近当前 28 天路线约束的一方，并把另一方标记为“待验证”。
- 若某结论很有道理但会显著增加 28 天范围，应默认列入“改写后并入”或“暂缓”。
- 汇总时必须始终回到三项判断：
  1. 是否直接服务 28 天主线；
  2. 是否帮助零基础学习者更稳完成闭环；
  3. 是否能在当前任务预算内转化为教程结构变更。

## 9. 结论

本轮梳理后，可以把 `zero-basics-plan` 的综述调研目标收敛为一句话：

> **围绕当前已成形的 28 天课程路线，优先调研那些能直接改善学习者边界定义、日更练习闭环、复盘机制、系统课串联与高风险报错处理的主题，而不是扩展到泛化零基础教育研究。**

据此，后续综述调研的主范围已经明确：

- 主范围不是“课程该教什么新主题”，而是“现有主线如何更稳落地”；
- 主重点不是“无限加资料”，而是“为课程结构补最关键的证据与模板”；
- 主任务不是“重写 28 天路线”，而是“为下一步资料检索、综述摘要和课程并入动作建立统一口径”。

这使得后续任务可以直接进入资料检索和证据归并阶段，而不必再反复讨论主题边界。
