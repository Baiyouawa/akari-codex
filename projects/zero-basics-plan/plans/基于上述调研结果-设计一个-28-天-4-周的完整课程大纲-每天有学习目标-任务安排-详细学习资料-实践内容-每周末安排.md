## Goal Description
基于已有调研结果，制定一套可执行的实施计划，用于产出一份面向“零基础学习者”的 28 天、4 周完整课程大纲，并最终整理为 Markdown 教程。

该任务的目标不是直接罗列课程内容本身，而是规划出一条清晰、可分工、可验收的执行路径，使 Multi-Agent 系统能够并行完成以下工作：

1. 对已有调研结果进行结构化梳理与去重。
2. 将课程范围拆分为 4 个递进式学习周与 28 个学习日。
3. 为每天生成统一格式的内容模块，包括：
   - 学习目标
   - 任务安排
   - 详细学习资料
   - 实践内容
4. 为每周末单独设计一次“系统课程”，用于回顾、串联和综合练习。
5. 将多来源资料统一为风格一致、难度连续、适合零基础用户的 Markdown 教程。
6. 建立质量检查机制，确保课程内容完整、节奏合理、资料可用、练习可执行、周与周之间衔接自然。

最终产物应是一份可直接交付、可继续细化、可用于后续生成正式教程文档的课程大纲方案与执行框架。

- AC-2: 每个学习日必须具备统一且完整的内容字段：学习目标、任务安排、详细学习资料、实践内容。
  - Positive Tests: 任意一天都能找到以上四类信息，字段表达清晰且可执行。
  - Negative Tests: 某些天仅有标题或只有任务列表；缺少资料来源说明；没有实践内容；字段命名和结构不统一。

- AC-3: 课程难度必须符合“零基础计划”定位，体现从入门到综合应用的渐进式路径。

  - Negative Tests: 第 1-3 天即要求复杂项目实战；知识点跳跃过大；缺乏前置铺垫；出现默认用户已具备基础能力的安排。

- AC-4: 课程设计必须显式利用“上述调研结果”，将多平台资料整合为统一学习路径，而非简单堆砌链接。
  - Positive Tests: 资料经过分类、去重、优先级排序，并说明为何放在对应天数与阶段。

  - Negative Tests: 周末课程仅重复前几天任务；没有综合练习；没有对本周知识结构进行串联。

- AC-7: 实施计划必须支持 Multi-Agent 并行执行，明确角色分工、依赖关系、合并节点和质检节点。

  - Negative Tests: 只有高层原则，没有实际可执行步骤；无法据此开始内容编写。

## Path Boundaries

### Upper Bound (Maximum Scope)
最完整的方案应包含以下内容：

1. 基于调研结果建立“课程知识地图”，明确核心主题、次级主题、先修关系与学习顺序。
2. 对多平台资料进行统一标签化处理，例如：
   - 入门概念
   - 基础操作
   - 重点难点
   - 案例演示
   - 练习参考
   - 延伸阅读
3. 产出完整的 28 天课程框架草案，并为每天填充：
   - 学习目标
   - 任务清单
   - 资料清单与使用顺序
   - 实践任务
   - 预计耗时
   - 当日交付物
4. 为每周末系统课程额外设计：
   - 本周知识回顾
   - 结构化总结图
   - 综合练习
   - 自测题或检查项
   - 复盘建议
5. 建立统一 Markdown 模板，包括标题层级、内容块命名、资料引用格式、提示框格式、练习与答案格式。
6. 明确 Multi-Agent 分工机制：
   - 调研整理 Agent
   - 课程架构 Agent
   - 每日内容编写 Agent
   - 练习设计 Agent
   - 审校与一致性 Agent
   - Markdown 统稿 Agent
7. 建立完整质检流程，包括：
   - 结构完整性检查
   - 难度递进检查
   - 资料有效性检查
   - 重复内容检查
   - 实践可执行性检查
   - 零基础适配性检查
8. 提供最终发布前的审阅清单，确保课程可直接进入写作或发布阶段。

1. 明确 4 周主题划分与每周目标。
2. 明确 28 天结构，其中每周最后一天为系统课程。
3. 为每天定义统一的四个字段：
   - 学习目标
   - 任务安排
   - 详细学习资料
   - 实践内容
4. 提供课程生成模板和内容组织规范。
5. 说明 Multi-Agent 的最小分工方式：
   - 一个代理负责资料整理
   - 一个代理负责课程结构设计
   - 一个代理负责质检与统稿
6. 提供最基本的验收标准，确保课程完整、连续、可转 Markdown。

## Dependencies and Sequence

### Milestones
1. 明确输入边界与目标规格
   - 汇总“上述调研结果”的来源范围、内容类型、主题覆盖度。
   - 明确课程对象为零基础学习者，确认课程最终形式为 28 天 Markdown 教程。
   - 定义课程必须包含的固定字段与周末系统课程机制。

2. 建立课程框架约束
   - 拆分为 4 周、28 天。
   - 为每周确定主线目标、难度等级与核心成果。
   - 确定每周系统课程的标准模板与功能边界。

3. 并行整理调研资料
   - 将已有资料按主题、难度、形式、平台来源进行分类。
   - 去除重复、低质量、过时或不适合零基础的内容。
   - 标记“必学资料”“选学资料”“补充资料”。

4. 设计知识递进路线
   - 确定先学什么、后学什么。
   - 建立天与天之间的前置依赖关系。
   - 避免知识跳跃和内容断层。

5. 产出周级大纲
   - 为 Week 1 到 Week 4 定义：
     - 周目标
     - 本周核心主题
     - 本周关键产出
     - 周末系统课程定位
   - 审核周级逻辑是否符合零基础递进。

6. 产出日级模板与编写规范
   - 明确每天固定包含：
     - 学习目标
     - 任务安排
     - 详细学习资料
     - 实践内容
   - 如有需要，可增加建议耗时、输出物、自测问题等辅助字段，但不得替代必选字段。

7. 并行编写 28 天课程草案
   - 由多个代理按周或按模块并行生成每日内容。
   - 保持术语、格式、难度标准一致。
   - 系统课程由专门代理或高级代理统一设计，避免周末质量不一致。

8. 统一合并与风格校准
   - 合并各代理产出的周内容。
   - 修正重复、冲突、格式不一、难度不平衡问题。
   - 统一 Markdown 标题层级、命名、资料引用与练习呈现方式。

9. 进行系统验收
   - 对照验收标准逐项检查。
   - 检查是否完整覆盖 28 天、是否每周有系统课程、是否每天四字段齐全。
   - 检查资料是否与当天目标匹配，练习是否可执行。

10. 输出最终可交付版本
   - 形成课程大纲主文档。
   - 附上模板、资料规范、审校清单。
   - 准备进入下一阶段的正式教程写作或发布。

## Implementation Notes
- 先统一课程模板，再开始多代理并行编写，避免后期大规模返工。
- 优先保证课程结构正确，其次再优化资料丰富度与练习多样性。
- 周级设计应先于日级设计完成，否则容易出现每日内容堆叠但整体节奏失衡。
- 每周系统课程必须单独设计，不能简单视为“复习日”。
- 每日“详细学习资料”应包含资料用途说明，例如：概念建立、操作演示、补充理解、案例参考。
- “实践内容”必须可执行、可验证，不能只写“自行练习”。
- 资料整合时应优先选择稳定、基础、讲解清晰的来源，而非信息密度高但门槛过高的资料。
- 不同代理产出的内容在合并前应经过统一字段检查与风格检查。
- 建议先产出课程骨架，再进行资料回填与实践设计，以提高并行效率。
- 如果调研结果存在空白主题，应以“最小必要补充”为原则填补缺口，并保留补充标记。
- 最终 Markdown 文档应采用固定标题层级，便于后续拆分为单日教程或整周教程。
- 代码中不应包含 AC-、Milestone 等计划术语
