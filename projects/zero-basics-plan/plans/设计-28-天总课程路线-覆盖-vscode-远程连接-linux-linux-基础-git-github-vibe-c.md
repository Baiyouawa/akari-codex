## Implementation Notes
- 先定义目标用户，建议默认对象为具备基础计算机使用能力、但对 Linux、Git、AI 编程工具和深度学习尚处于入门阶段的学习者
- 先做“课程地图”，再填“日计划细节”，避免先写细节导致整体结构失衡
- 主题依赖建议为：环境与远程连接 → Linux 基础 → Git/GitHub → AI 编程工作流（Vibe Coding、Cursor）→ 基础深度学习 → 综合复盘
- 周结构建议采用“5 天学习 + 1 天实践巩固 + 1 天系统课程”或“6 天推进 + 1 天系统课程”的固定节奏，以保证节奏稳定
- 每周系统课程应承担三个作用：阶段复盘、知识串联、综合练习
- 第一周重点应放在环境搭建、VSCode 远程连接 Linux、Linux 基础命令与文件系统认知
- 第二周重点应放在 Linux 实战深化与 Git/GitHub 基础协作能力建立
- 第三周重点应放在 Vibe Coding 与 Cursor 的工作流理解、提示方式、代码辅助实践和常见风险
- 第四周重点应放在基础深度学习概念、最小实验、与前述开发环境和版本管理能力结合
- 基础深度学习范围建议控制在：张量/数据表示、训练与推理概念、常见网络直觉、简单框架体验、最小分类任务或推理示例，不建议扩展到复杂数学推导或大规模训练部署
- 多平台调研时，建议按“官方权威性、适合初学者程度、实践可操作性、内容时效性、是否可复现”进行评分
- 资料汇总时，优先保留稳定知识与高频操作路径，避免纳入过多依赖特定版本、特定收费平台或短期热点工具特性的内容
- Markdown 文档建议包含以下章节：课程说明、学习前准备、四周总览、每日计划、每周系统课程、实践建议、参考资料
- 每日计划建议至少包含：主题、学习目标、核心知识点、练习任务、建议输出物
- 建议为每周设计一个可见成果，例如：第 1 周完成远程开发环境连通，第 2 周完成仓库协作练习，第 3 周完成 AI 辅助编码小任务，第 4 周完成基础深度学习最小项目体验
- 若为多人协作，建议按主题拆分调研责任，再由一人统一术语、格式与文风，避免拼接感过强
- 文档评审时，重点检查：是否真有 28 天、是否每周末都有系统课程、是否六大主题全部覆盖、是否适合初学者循序上手
- 最终交付应以 Markdown 为主，不强制要求代码仓库、幻灯片、视频或录播内容
- 代码中不应包含 AC-、Milestone 等计划术语
