# 对现有 28 天整合稿做缺口扫描：标记哪些天缺少操作步骤、来源链接、练习或复盘作业，并给出补全建议

Generated: 2026-03-26

## Goal Description
对现有 28 天整合稿逐天进行系统性缺口扫描，建立一份可执行的缺口清单，明确每一天是否完整包含以下四类核心要素：操作步骤、来源链接、练习任务、复盘作业。对缺失项进行标记、分类和优先级判断，并同步提出可直接落地的补全建议，确保后续能够并行补查资料、补写内容并最终形成一份结构完整、标准统一、可交付的 28 天文档。

该任务的重点不只是“找出缺什么”，还包括：
1. 形成统一的扫描标准，避免不同天采用不同判断口径。
2. 标记缺失严重程度，区分“完全缺失”“部分缺失”“表达不清”“疑似缺失待核实”。
3. 给出补全建议时尽量具体到可执行动作，例如补哪类步骤、补什么类型链接、补哪种练习形式、复盘作业应包含哪些问题。
4. 支持并行推进，即扫描与补查可分轨进行，但最终输出需要可汇总、可追踪、可验收。

## Acceptance Criteria

- AC-1: 已建立统一的 28 天扫描标准，并按同一口径覆盖全部 28 天内容
  - Positive Tests: 对 Day 1 至 Day 28 均有扫描记录；每一天都按“操作步骤 / 来源链接 / 练习 / 复盘作业”四项进行检查；判定标签格式一致
  - Negative Tests: 只扫描部分天数；有的天检查四项，有的天只检查两三项；同类问题在不同天使用不同标准导致无法横向比较

- AC-2: 每一天的缺口状态都有清晰标记，能区分完整、部分缺失、完全缺失、待核实
  - Positive Tests: 某天缺少练习但有复盘时被标记为“练习缺失”；某天只有概念说明没有可执行步骤时被标记为“操作步骤缺失”；某天引用了资料但无可访问链接时被标记为“来源链接缺失或不完整”
  - Negative Tests: 仅笼统写“内容不完整”；无法判断究竟缺哪一类；把“质量差”与“缺失”混为一谈

- AC-3: 对每个缺失项都提供补全建议，且建议具备可执行性
  - Positive Tests: 对缺少操作步骤的天，给出建议如“补充按序编号的执行流程，至少包含准备条件、执行动作、产出标准”；对缺少来源链接的天，建议“补一手来源或高可信二手来源，并附可访问 URL”；对缺少练习或复盘的天，给出具体练习题型或复盘问题模板
  - Negative Tests: 建议仅写“后续补充即可”“需要完善”；没有说明补什么、怎么补、补到什么程度

- AC-4: 输出结果支持并行补查与后续跟踪
  - Positive Tests: 结果中包含按天汇总的缺口清单、缺口类型、补全建议、优先级或处理顺序；可以让不同人员分别认领“补链接”“补练习”“补复盘”等任务
  - Negative Tests: 输出是不可拆分的大段文字；无法按天或按缺口类型分配；缺少状态字段导致后续不能跟踪

- AC-5: 扫描结论能够反映文档完整性风险，帮助优先处理高风险天数
  - Positive Tests: 能识别“同时缺少步骤、链接、练习、复盘”的高风险天；能将“仅缺复盘题”的低风险天与“仅有标题无正文”的高风险天区分开
  - Negative Tests: 所有缺口一视同仁；无法判断优先处理顺序；未识别关键性缺口对整体交付的影响

- AC-6: 最终产出可直接作为补齐工作的输入材料
  - Positive Tests: 扫描结果能被直接转为补写任务清单；每一天都有明确的“现状—缺口—建议”结构；补全人员无需二次猜测任务含义
  - Negative Tests: 扫描报告偏描述性、不可执行；补齐人员仍需重新阅读整份文档才能理解问题

## Path Boundaries

### Upper Bound (Maximum Scope)
最完整的方案包括：
1. 逐天检查 28 天整合稿全部内容，按统一模板记录四项要素是否齐备。
2. 对每一天输出：
   - 当前内容摘要
   - 四项要素状态
   - 缺失描述
   - 严重程度
   - 补全建议
   - 补充资料方向
   - 建议优先级
3. 识别跨天共性问题，例如：
   - 多天都缺来源链接
   - 多天只有知识说明、缺少操作步骤
   - 练习与复盘设计标准不统一
4. 形成汇总视图：
   - 缺口天数统计
   - 各缺口类型分布
   - 高风险天数列表
   - 可并行处理的补齐分组
5. 对“待核实”内容建立补查路径，说明应优先查找一手资料、原始出处、平台官方说明或高可信补充来源。
6. 给出补齐标准模板，便于后续统一补写风格，例如：
   - 操作步骤建议至少 3—5 步
   - 每天至少 1 个来源链接
   - 每天至少 1 个练习任务
   - 每天至少 1 组复盘问题或作业要求

### Lower Bound (Minimum Scope)
最简可行方案包括：
1. 覆盖全部 28 天，不遗漏。
2. 仅检查四项核心要素：操作步骤、来源链接、练习、复盘作业。
3. 对每一天给出缺失标记。
4. 对存在缺失的天提供一句以上明确补全建议。
5. 输出一份按天排列的缺口清单，能够直接指导后续补齐工作。

### Allowed Choices
- Can use: 现有 28 天整合稿全文、既有项目记录、此前失败记录中的问题点、统一的扫描表格或检查清单、按天编号的缺口标记方式、来源可信度分级、并行补查分工表
- Can use: “完整 / 部分缺失 / 完全缺失 / 待核实”等状态标签
- Can use: 按缺口类型拆分任务，如补步骤、补链接、补练习、补复盘
- Cannot use: 跳过未完成天数后直接假定完整
- Cannot use: 未核实即补写虚构来源链接
- Cannot use: 只凭印象判断某天“应该有内容”而不回看原文
- Cannot use: 把质量优化项当作缺失项混入核心缺口清单但不作区分
- Cannot use: 输出无法追踪的自由文本结论，导致后续无法分配和验收

## Dependencies and Sequence

### Milestones
1. 明确扫描口径与输出模板
   - 定义四项核心检查对象：操作步骤、来源链接、练习、复盘作业
   - 约定状态标签与严重程度规则
   - 确定逐天记录格式，避免后续返工

2. 获取并锁定扫描对象
   - 确认当前版本的 28 天整合稿为唯一检查基线
   - 收集前面失败记录中的已知问题，作为重点复核提醒
   - 若存在多个版本，先完成版本对齐，避免扫描结果失真

3. 逐天执行缺口扫描
   - 按 Day 1 至 Day 28 顺序检查
   - 每天分别判断四项是否存在、是否可用、是否表达清晰
   - 标记“完全缺失 / 部分缺失 / 待核实 / 完整”

4. 形成逐天缺口清单
   - 汇总每一天缺什么
   - 记录证据或判断依据，如“只有理论介绍，无步骤编号”“提到参考资料但无 URL”
   - 初步标注优先级

5. 给出补全建议
   - 按缺口类型为每一天生成补全建议
   - 建议尽量模板化，便于并行补齐
   - 对待核实项给出补查方向和推荐来源类型

6. 汇总跨天问题与并行处理方案
   - 统计哪些缺口最常见
   - 识别可批量补齐的内容类型
   - 给出优先处理顺序，例如先补高风险天，再补统一性问题

7. 输出最终交付物
   - 生成按天缺口总表
   - 生成高风险清单
   - 生成补全建议汇总
   - 确保结果可直接用于认领与补齐

## Implementation Notes
- 建议先设计统一检查表头，例如：Day、操作步骤状态、来源链接状态、练习状态、复盘状态、缺口说明、补全建议、优先级、备注
- “操作步骤”应重点检查是否具备可执行性，而非仅有概念描述；若只有原则、没有动作序列，应判为缺失或部分缺失
- “来源链接”应检查是否真实存在、是否可访问、是否能支撑当天内容；仅写“参考某书/某文章”但无定位信息时，通常不能视为完整
- “练习”应检查是否有明确任务指令、输入输出要求或完成标准；仅写“自行练习”不应视为完整
- “复盘作业”应检查是否存在复盘问题、总结要求、提交产物或回顾模板；只有一句“记得复盘”通常不算完整
- 建议为缺口设置严重程度，例如：高（影响当天可执行性）、中（影响学习闭环）、低（影响完整性但不阻断执行）
- 对“待核实”项应单独标注，不要直接并入“完整”；待核实通常适用于提到来源但链接失效、内容疑似有练习但表达不明确等情况
- 可以采用“两轮法”提高效率：第一轮快速扫 28 天找缺口，第二轮集中核实边界项并完善建议
- 并行补查时可按缺口类型分工：一组补来源链接，一组补练习与复盘，一组补操作步骤，最后统一合稿
- 若某一天同时缺 3 项及以上，建议列入高优先级，优先补齐
- 若某一天四项齐全但质量偏弱，可记录为“优化项”，但应与“缺失项”分开管理
- 最终输出中应确保每一天都有结果，即使内容完整也要明确写明“已检查，无核心缺口”
- 代码中不应包含 AC-、Milestone 等计划术语
