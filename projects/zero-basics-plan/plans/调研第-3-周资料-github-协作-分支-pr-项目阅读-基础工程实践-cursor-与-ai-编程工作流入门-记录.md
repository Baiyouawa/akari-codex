## Implementation Notes
- 建议先以“主题 > 子主题 > 资料卡片”的三级结构整理，避免先堆链接后返工
- 推荐为每条资料分配唯一编号，便于在教程大纲中引用
- 可使用统一字段模板：标题｜链接｜平台｜作者/机构｜时间｜访问日期｜主题｜摘要｜适合人群｜推荐等级｜可信度｜备注
- 对 GitHub 协作与分支/PR，优先采用官方文档和真实开源仓库示例交叉验证
- 对项目阅读主题，应同时收集“如何阅读代码库的方法论”和“真实项目案例”两类资料
- 对基础工程实践，避免过度扩展到完整 DevOps 体系，应聚焦初学者在教程中最需要掌握的最小工程习惯
- 对 Cursor 与 AI 编程工作流，应明确区分“工具介绍”“操作流程”“最佳实践”“风险与限制”
- AI 相关资料需要额外记录版本、发布日期与适用前提，避免将快速过时内容直接写入教程正文
- 如发现不同来源观点冲突，应保留冲突说明，并优先采用官方或可验证实践
- Markdown 文档中可使用表格汇总资料，也可在主题下用条目列出重点摘要，但应保持全篇格式一致
- 输出内容应兼顾“可读性”和“可复用性”，既方便团队讨论，也方便后续直接转写为教程章节
- 可以附加“缺口清单”，记录当前尚未找到高质量资料的子主题，便于下一轮补充
- 若多人并行调研，建议按主题拆分文件并在总索引中汇总，最后再统一合并
- 需要保留原始链接，不要只保留平台搜索页或截图
- 对引用开源项目内容时，应优先链接到具体仓库页面、官方文档页或 CONTRIBUTING/README 文件
- 最终交付前至少进行一次人工抽检，确认摘要与原文一致，不存在误读或过度概括
- 代码中不应包含 AC-、Milestone 等计划术语
