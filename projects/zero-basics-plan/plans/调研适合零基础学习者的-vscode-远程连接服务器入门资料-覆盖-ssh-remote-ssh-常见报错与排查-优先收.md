## Implementation Notes
- 先定义统一的数据结构，再开始多平台并行检索，避免后期汇总返工
- 优先判断“是否适合零基础”，再判断“信息是否全面”，避免被高质量但过于进阶的内容占据主要位置
- 对每条资料建议增加“使用场景”字段，如首次入门、连接失败排查、Windows 用户专用、云服务器用户专用
- 对小红书、知乎等平台内容要注意信息碎片化问题，可通过“主帖 + 评论区补充”的方式摘录，但必须标明信息来源范围
- 对博客类内容要识别作者是否有实际使用经验，优先保留包含命令截图、错误日志、排查过程的实战文章
- 官方文档可用于校正插件名称、参数说明、版本差异，但不应作为主体推荐内容
- 对报错排查类内容，建议按“现象—可能原因—解决办法—参考来源”结构沉淀，方便后续教程复用
- 对于明显依赖特定云厂商、特定 Linux 发行版或公司内网环境的教程，应标注适用限制
- 建议使用“必读 / 选读 / 备用”三级优先级，帮助后续 28 天课程编排
- 输出结果中应区分“适合直接引用的资料”和“仅适合内部参考的资料”
- 代码中不应包含 AC-、Milestone 等计划术语
