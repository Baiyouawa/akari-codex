  - Positive Tests: 每条资料包含标题、来源平台、原始链接、作者/机构、发布时间或版本信息、主题标签、适用难度、可教学使用说明、备注

- AC-3: 完成多平台资料质量评估并形成筛选结论

- AC-4: 输出内容能直接支持教程大纲与 Markdown 文档编写

## Path Boundaries

### Upper Bound (Maximum Scope)
完成五大主题的全量系统调研，覆盖官方文档、知名教程网站、中文技术社区、英文技术资料、视频课程、代码仓库、问答社区与实验环境示例；对每个主题建立“核心主资料 + 辅助资料 + 实战案例 + 常见误区 + 练习资源 + 延伸阅读”六层结构；为每个知识点提炼教学摘要、推荐命令、演示顺序、常见坑点与适合放入 Markdown 的内容块模板；同时输出去重后的资料台账、优先级排序、使用级别、风险备注、引用格式建议和课程映射表，以便直接进入第 2 周教程编写阶段。

### Lower Bound (Minimum Scope)
至少完成五大主题的基础资料收集与初筛；每个主题至少筛出 3 份可信资料，其中至少 1 份可直接教学使用；每条资料记录标题、来源、链接、主题、难度、适用性说明；给出按教学顺序整理的主题知识点清单与推荐主参考来源，确保后续能据此开始编写第 2 周 Markdown 教程初稿。

### Allowed Choices
- Can use: 官方文档、Linux 发行版文档、Git 官方文档、OpenSSH 官方资料、知名技术社区文章、优质博客、教程网站、公开视频课程、示例仓库、命令手册、团队共享文档、Markdown 表格或清单形式归档
- Can use: 中文与英文资料并行调研，并在记录中补充适用对象与语言说明
- Can use: 以教学可用性、准确性、时效性、可复现性、结构完整性作为主要筛选维度
- Can use: 将资料按“直接可教 / 需整理改写 / 仅参考”分级
- Cannot use: 无法确认来源的转载内容、失效链接、盗版课程、版权风险明显的封闭内容搬运、缺少关键上下文的碎片化问答作为主资料

## Dependencies and Sequence

### Milestones
1. 明确第 2 周范围与调研口径
   - 确认五大主题的边界：Linux 进阶、文件权限、进程管理、SSH/服务器常用操作、Git 基础与实战
   - 统一资料记录模板、命名规则、标签体系、难度分级与使用级别标准
   - 明确并行分工方式与合并节点，避免多人重复调研
2. 建立资料源清单与优先级
   - 优先列出官方文档与高可信平台
   - 补充中文教学友好资料、英文权威资料、视频与案例仓库
   - 按主题分配主搜集源与备选源
3. 并行开展主题资料搜集
   - Linux 进阶：目录结构、常用文本处理、包管理、环境变量、管道与重定向等
   - 文件权限：用户与用户组、rwx、chmod、chown、umask、特殊权限
   - 进程管理：ps、top、htop、jobs、kill、systemd/service、前后台任务
   - SSH/服务器常用操作：ssh、scp、rsync、密钥登录、known_hosts、常见远程维护命令
   - Git 基础与实战：仓库初始化、提交、分支、合并、冲突、远程仓库、常见协作流程
4. 对资料进行初筛与去重
   - 剔除失效、过时、重复、低质量或不适合教学的资料
   - 标记可直接使用、需整理、仅参考三类
   - 保留淘汰原因，方便团队复核
5. 完成质量评估与教学适配标注
   - 检查概念准确性、命令可复现性、章节结构清晰度、示例质量、对初学者友好程度
   - 为每条资料补充教学摘要、推荐使用场景和注意事项
   - 为高优先级资料补充引用方式与对应课程位置
6. 构建第 2 周教学映射表
   - 将资料与每天/每节课的知识点建立对应关系
   - 给出建议教学顺序、演示案例和练习方向
   - 标出仍需补充或自编写的空白点
7. 输出可交付归档结果
   - 形成统一 Markdown 或表格文档
   - 包含资料总表、主题分表、推荐主资料列表、风险备注、待补项
   - 完成团队复核并提交用于 28 天教程大纲编写

## Implementation Notes
- 先建立统一资料卡片字段：标题、主题、子主题、来源平台、作者/机构、链接、语言、发布时间/版本、难度、内容类型、使用级别、教学摘要、推荐章节、备注
- 建议采用“主题目录 + 总表索引”的双层结构，便于按主题查看与全局检索
- 每个主题优先选择 1 份官方/权威主资料，再搭配 2 至 5 份更适合教学讲解的辅助资料
- 对直接教学使用的资料，需额外记录其适合用于概念讲解、课堂演示、课后练习还是案例实战
- 对命令类内容至少做一次基础可用性检查，避免将明显错误或依赖特殊环境的示例纳入主教程
- Git 相关资料需特别关注命令行为与当前版本一致性，并区分本地工作流、远程协作、分支管理等场景
- SSH/服务器常用操作资料需注意安全性，不收录鼓励弱密码、关闭校验或高风险配置的示例作为推荐方案
- 文件权限与进程管理模块应补充常见误区，如 sudo 使用边界、递归修改权限风险、误杀进程、后台任务丢失等
- Linux 进阶部分应优先选择兼容主流发行版的通用内容，发行版特有命令需单独备注
- 记录来源时优先保留原始发布页，不优先使用转载页或二次整理页
- 若资料为视频或课程，需补充时间戳、对应知识点和是否需要二次转写整理
- 若资料为英文内容，建议附一句中文适用说明，帮助后续编写中文教程时快速判断可用性
- 团队协作时应设定去重规则，如“同一原始链接唯一保留、转载合并到原始来源、同主题同知识点保留质量更高版本”
- 最终归档应突出“可直接进入教学文档”的优先资料，避免输出成为单纯的资料收藏夹
- 代码中不应包含 AC-、Milestone 等计划术语
