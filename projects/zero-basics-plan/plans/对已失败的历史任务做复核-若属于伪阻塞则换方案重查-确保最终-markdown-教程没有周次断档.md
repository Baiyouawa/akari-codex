# 对已失败的历史任务做复核，若属于伪阻塞则换方案重查，确保最终 Markdown 教程没有周次断档

Generated: 2026-03-26

## Goal Description
对历史上已标记为失败的任务逐项复核，识别其中因方法不当、检索路径单一、误判条件、资源选择失误等造成的“伪阻塞”情况；对可继续推进的任务立即切换备选方案并并行补查缺失内容；最终完成一份覆盖连续 28 天、按周次组织且不存在断档的 Markdown 教程文档。实施过程中需要保留复核依据、替代方案选择逻辑、缺失补齐状态与最终合并结果，确保小白能够自行兜底完成交付。

## Acceptance Criteria

- AC-1: 所有历史失败任务均已完成复核，并形成明确分类结论
  - Positive Tests: 每条失败记录均有唯一标识、失败原因、是否属于伪阻塞、复核结论、后续动作；不存在“未判断”状态残留
  - Negative Tests: 仍有失败任务未被复核；仅保留“失败”标签但无复核说明；同一任务存在相互矛盾的结论
- AC-2: 被判定为伪阻塞的任务均已采用替代方案重新检查
  - Positive Tests: 每个伪阻塞任务至少记录一种替代检索/补查方案，并有执行结果；替代方案与原失败路径不同
  - Negative Tests: 判定为伪阻塞后未重新执行；重新检查仍沿用原失败路径且无任何变化；无结果留痕
- AC-3: 缺失内容已并行补齐，并覆盖完整 28 天周期
  - Positive Tests: 1—28 天内容均存在，且能映射到对应周次；缺失天数清单为空；并行补查任务有状态同步机制
  - Negative Tests: 任意一天内容缺失；出现重复天数替代缺失天数；补查状态分散无法确认是否补齐
- AC-4: 最终 Markdown 教程按周次连续组织，不存在周次断档或结构跳跃
  - Positive Tests: 教程中周次顺序连续，例如第 1 周至第 4 周完整衔接；每周下属天数齐全；目录与正文一致
  - Negative Tests: 缺少某一整周或部分周；周次编号跳号；目录存在但正文缺失；正文存在但周次归属不明
- AC-5: 最终文档内容来源、补齐方式与复核结果可追溯
  - Positive Tests: 每段补齐内容可回溯到原任务、复核结论或替代方案；有版本对照或变更记录
  - Negative Tests: 无法区分原始成功内容与后补内容；内容已补入但无来源说明；复核结果与最终文档无法关联
- AC-6: 交付物可直接作为完整 Markdown 教程使用
  - Positive Tests: Markdown 结构规范，标题层级一致，列表/代码块/引用等格式正确；可正常渲染和阅读
  - Negative Tests: 格式破损、标题层级混乱、存在明显占位符未清理；需再次人工大规模整理后才能使用

## Path Boundaries

### Upper Bound (Maximum Scope)
对全部历史失败记录进行系统复核，建立“失败原因—伪阻塞判断—替代方案—补查结果—文档落点”全链路台账；并行使用多种替代路径补齐全部缺失内容；完成 28 天教程的结构统一、内容补全、周次重排、目录生成、交叉校验、格式修正、版本留痕和最终验收；同时输出一份简明复盘，说明高频伪阻塞模式及后续规避建议。

### Lower Bound (Minimum Scope)
仅对会影响 28 天完整性的失败任务做快速复核；将明显属于伪阻塞的任务改用至少一种备选方案重查；优先补齐所有缺失天数；合并成一份按周连续、无断档、可渲染的 Markdown 教程文档。

### Allowed Choices
- Can use: 历史失败记录表、任务日志、已有成功内容、备选检索路径、并行补查、人工复核、结构化清单、Markdown 模板、版本对照表、缺失映射表、周次校验表
- Cannot use: 跳过复核直接补写、仅凭主观印象判定伪阻塞、保留未确认缺口进入最终文档、打乱原有 28 天顺序、用占位符冒充完成内容、输出存在周次断档的 Markdown 成品

## Dependencies and Sequence

### Milestones
1. 汇总历史失败任务与现有 28 天内容现状
2. 建立失败任务复核清单，标注任务编号、原失败原因、影响范围、关联天数/周次
3. 定义伪阻塞判定规则，如渠道单一、关键词偏差、检索入口错误、格式误判、资源可替代但未尝试等
4. 逐项复核失败任务，输出“真实阻塞 / 伪阻塞 / 待补证据”结论
5. 为伪阻塞任务分配替代方案，并启动并行补查
6. 将补查结果映射回 1—28 天内容框架，更新缺失清单
7. 合并已有内容与新补内容，按周次重组 Markdown 教程结构
8. 执行完整性校验：天数连续性、周次连续性、目录正文一致性、格式正确性
9. 修正冲突、去重、补漏，形成最终 Markdown 教程版本
10. 输出复核结果摘要与最终交付确认

## Implementation Notes
- 先做“失败记录—天数—周次”三维映射，避免补查后仍无法定位到最终文档落点
- 对“伪阻塞”给出可执行判据，避免因定义模糊造成重复争议
- 替代方案应与原失败路径有实质差异，例如更换检索关键词、来源入口、拆分任务粒度、调整验证方式
- 并行补查时应设置统一状态字段，如“待复核、复核中、伪阻塞重查中、已补齐、确认无解”
- 若某任务被判定为真实阻塞，应同步确认其是否影响 28 天完整性；若影响，则必须设计替代性补齐策略，而非直接留空
- 合并文档前先生成 28 天覆盖矩阵，逐格确认是否已有内容、是否新补、是否待确认
- 周次建议固定映射为 4 周 × 7 天，防止出现周次重叠、跨周错位或编号跳跃
- 最终 Markdown 应统一标题层级、编号规范、列表样式与代码块格式，避免因拼接导致阅读断裂
- 对补齐内容增加轻量变更标记或内部追踪信息，但不要污染最终面向读者的正文表达
- 在最终校验中至少检查：缺失天数是否为零、周次是否连续、目录是否可跳转、正文是否存在空标题
- 若多条失败任务指向同一缺失天数，应先做去重整合，再执行补查，避免重复劳动
- 代码中不应包含 AC-、Milestone 等计划术语
