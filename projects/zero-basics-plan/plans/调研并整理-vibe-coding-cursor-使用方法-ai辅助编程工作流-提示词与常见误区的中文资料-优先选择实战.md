## Goal Description
围绕“零基础计划”的内容建设需求，系统调研并整理以下四类中文资料：vibe coding、Cursor 使用方法、AI 辅助编程工作流、提示词与常见误区。资料筛选以“实战型、可复现、适合初学者”为优先，最终形成可直接用于后续“28 天完整 Markdown 教程”编写的结构化素材库。

产出应包含：
1. 中文资料清单，覆盖四大主题。
2. 每条资料的标题、作者/来源平台、链接、发布时间（如可获取）、内容类型、适用人群、核心结论。
3. 对资料进行质量分级与用途标注，如“入门必读”“操作演示”“案例拆解”“误区提醒”。
4. 提炼可复用内容，为后续 28 天教程提供章节素材、示例思路与推荐阅读。
5. 明确剔除纯概念空谈、无实操、无链接来源或高度重复的内容。

## Acceptance Criteria

- AC-1: 完成四大主题的中文资料调研与归档，且每个主题都有可用内容
  - Positive Tests: 四个主题均至少收集到若干条中文资料；每个主题都有可用于教程编写的实战内容；资料条目能按主题清晰分类
  - Negative Tests: 仅覆盖其中 1-2 个主题；某一主题只有零散观点而无实操资料；所有资料混在一起无法分类

- AC-2: 每条入选资料均记录基础元信息与来源链接
  - Positive Tests: 每条资料包含标题、来源平台/作者、链接、内容摘要；链接可点击访问；能追溯原始出处
  - Negative Tests: 只有标题没有链接；只有平台名没有具体来源；引用二手搬运内容但未标明原始出处

- AC-3: 资料筛选优先实战型、适合零基础学习
  - Positive Tests: 入选内容包含操作步骤、案例演示、工作流拆解、提示词示例、常见报错或误区说明；能直接支撑教学
  - Negative Tests: 大量收录纯观点评论、行业趋势文章、营销稿、泛泛介绍；内容过于面向资深开发者且对初学者帮助有限

- AC-4: 对资料进行质量评估与用途标注，便于后续教程编排
  - Positive Tests: 每条资料带有“推荐级别/适用场景/适用阶段”等标签；能够区分“入门阅读”“进阶补充”“案例参考”“避坑必看”
  - Negative Tests: 资料只是简单罗列；没有筛选优先级；无法判断哪些资料适合纳入 28 天教程

- AC-5: 输出结果可直接支持“28 天完整 Markdown 教程”的大纲映射
  - Positive Tests: 资料整理结果中包含主题建议、可拆分章节、可插入示例、推荐学习顺序；能映射为按天安排的素材池
  - Negative Tests: 资料虽多但无法用于课程结构设计；没有主题归纳；无法区分先学后学顺序

- AC-6: 明确记录常见误区与不推荐做法
  - Positive Tests: 至少汇总一组与 AI 辅助编程相关的常见误区，如过度依赖一轮生成、缺乏验证、不会拆任务、提示词模糊等；每条误区附参考来源或实践依据
  - Negative Tests: 只收集“怎么做”，没有“不要怎么做”；误区内容完全主观且无来源支撑

- AC-7: 整理结果采用统一 Markdown 结构，便于并行协作和后续扩写
  - Positive Tests: 输出文档字段一致、层级清晰、可复制到知识库或教程仓库；同类资料格式统一
  - Negative Tests: 不同主题使用不同格式；字段缺失严重；无法直接纳入教程编写流程

## Implementation Notes
- 先定义“什么算实战型内容”，建议标准为：有操作步骤、有真实案例、有提示词示例、有工具界面或执行结果说明，至少满足其中两项
- 资料记录字段建议统一为：主题、标题、作者/来源、平台、链接、发布日期、内容类型、适用对象、实战程度、摘要、可复用点、推荐等级、备注
- 可将资料类型细分为：文章、视频、图文教程、案例复盘、问答帖、官方文档解读、工作流模板
- 推荐等级可使用简单分层：
  - A：零基础主线必读，可直接用于教程正文
  - B：适合作为案例、补充说明或延伸阅读
  - C：信息有参考价值，但不宜作为主线材料
- 对 Cursor 使用方法，优先收集以下子主题：
  - 安装与初始配置
  - Chat / Agent / Edit 等核心功能
  - 代码生成、重构、问答、项目级理解
  - 与终端、Git、测试、调试配合的实际流程
  - 适合初学者的最小可行用法
- 对 vibe coding，重点区分“概念解释”与“真实实践”，优先保留：
  - 从想法到原型的快速迭代案例
  - 使用 AI 进行页面、脚本、小工具开发的全过程记录
  - 对“氛围式开发”边界、风险、适用场景的说明
- 对 AI 辅助编程工作流，优先收集：
  - 需求拆解 → 提示词组织 → 生成代码 → 运行验证 → 修复迭代 → 提交整理
  - 面向零基础/非科班用户的工作方式
  - 小项目或真实任务的完整链路
- 对提示词与常见误区，建议按“好提示词特征 / 常见坏提示词 / 修正方式”整理，并优先纳入编程场景示例
- 对来源链接，优先记录原始发布页；如平台内容可能失效，可在备注中补充镜像、作者主页或存档信息
- 对视频类资料，建议补充时间戳摘要或关键章节，避免后续二次查找成本过高
- 对问答类资料，只有在回答内容结构完整、实操性强、可信度较高时才纳入正式清单
- 可在整理阶段同步输出两层结果：
  - 一层为“完整资料库”
  - 一层为“零基础计划优先阅读清单”
- 对明显存在时效性的资料，如 Cursor 界面、功能命名、定价、插件兼容信息，需要加注“版本时效注意”
- 若同一主题出现大量重复观点，应提炼为“共识总结”，并保留最有代表性的 2-3 个来源
- 最终 Markdown 文档建议按以下顺序组织：总览说明 → 主题一 → 主题二 → 主题三 → 主题四 → 常见误区 → 推荐阅读顺序 → 28 天教程映射
- 代码中不应包含 AC-、Milestone 等计划术语
