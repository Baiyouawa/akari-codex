## 分阶段执行计划

## Phase 1：问题定义与范围收敛（零资源）

1. 明确本项目中的 “MoE” 指代为 Mixture-of-Experts 模型/系统研究与工程工作空间，而非泛化缩写。
2. 产出一份问题定义文档，覆盖：
   - 核心概念：expert、router、capacity、load balancing、active parameters
   - 研究对象：模型结构、训练稳定性、推理效率、部署成本、评测方法
   - 非目标：当前阶段不直接承诺训练大模型或调用外部闭源 API
3. 建立开放问题列表，例如：
   - 本项目更偏研究综述、复现实验，还是实现工作台？
   - 首个里程碑是写作性产物、代码原型，还是 benchmark 方案？

预期产出：`README.md`、`TASKS.md`、后续可新增 `analysis/` 或 `literature/` 文档。

## Phase 2：资料与任务树初始化（零资源）

1. 梳理 MoE 相关资料收集任务：经典论文、近期工作、工程实现、常见失败模式。
2. 梳理评估任务：
   - 质量/能力：相对 dense baseline 的效果收益
   - 效率：激活参数、吞吐、延迟、显存/训练成本
   - 稳定性：路由塌陷、负载不均、专家失活
3. 梳理实现方向任务：
   - 最小可运行 toy MoE
   - 路由/负载均衡机制对比
   - 推理与训练观测脚本

预期产出：结构化任务清单。

## Phase 3：进入执行前的决策门（可能资源相关）

在完成 Phase 1-2 后，再决定是否进入以下任一方向：

1. 文献综述优先：形成系统性文献地图。
2. 原型实现优先：在本仓库内创建最小 MoE 实验代码/脚本。
3. 评测设计优先：制定 dense vs MoE 的对比方案。

决策依据：
- 哪条路径在当前仓库约束下能产出最多可验证知识
- 是否需要额外预算、数据或外部模型访问

## 建议的近期执行顺序
